CONTROL DE SUSCRIPCIONES (PLAN DE IMPLEMENTACION)

Objetivo
- Mantener la app principal local (backend + frontend en la clinica).
- Controlar acceso de forma centralizada con una base de datos remota de suscripciones.
- Poder bloquear/desbloquear clinicas sin tocar cada instalacion local.

Arquitectura propuesta
- App local:
  - Backend Node/Express (actual proyecto).
  - Frontend servido por Express (actual proyecto).
- Base de datos local:
  - Usuarios, pacientes, agenda, cobros, etc. (actual).
- Base de datos remota (Railway MySQL):
  - Solo tablas de licenciamiento/suscripcion.
- Validacion:
  - El backend local consulta la BD remota al iniciar sesion.
  - Si suscripcion no activa, deniega acceso.

Flujo funcional (login)
1. Usuario envia credenciales.
2. Backend valida usuario/clave local.
3. Backend valida suscripcion remota por clinica_id.
4. Si estado es invalido o vencida -> HTTP 403.
5. Si activa -> login normal y token.

Estados de suscripcion sugeridos
- activa
- suspendida
- vencida
- cancelada

Modelo minimo en BD remota
Tabla: suscripciones
- idSuscripcion (PK, bigint autoincrement)
- clinicaId (varchar(64), unique)
- estado (varchar(20))
- fechaInicio (datetime)
- fechaVencimiento (datetime)
- planNombre (varchar(50))
- maxUsuarios (int null)
- observacion (varchar(255) null)
- updatedAt (datetime)

SQL base sugerido
CREATE TABLE suscripciones (
  idSuscripcion BIGINT AUTO_INCREMENT PRIMARY KEY,
  clinicaId VARCHAR(64) NOT NULL UNIQUE,
  estado VARCHAR(20) NOT NULL,
  fechaInicio DATETIME NULL,
  fechaVencimiento DATETIME NULL,
  planNombre VARCHAR(50) NULL,
  maxUsuarios INT NULL,
  observacion VARCHAR(255) NULL,
  updatedAt DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

CREATE INDEX idx_suscripciones_estado ON suscripciones(estado);
CREATE INDEX idx_suscripciones_vencimiento ON suscripciones(fechaVencimiento);

Regla de decision recomendada
- Permitido solo si:
  - estado = 'activa'
  - y (fechaVencimiento is null o fechaVencimiento >= NOW()).

Manejo de caida de internet (grace period)
- Guardar cache local del ultimo resultado valido:
  - estadoUltimoCheck
  - vencimientoUltimoCheck
  - ultimaValidacionAt
- Si remoto no responde:
  - Permitir acceso solo durante ventana de gracia (ej: 72 horas)
  - Despues de ese tiempo: bloquear login hasta revalidar.

Tabla local cache sugerida
Tabla: licencia_cache
- id (PK tinyint = 1)
- estado (varchar(20))
- fechaVencimiento (datetime null)
- ultimaValidacionAt (datetime)
- fuente (varchar(20)) -- 'remoto' o 'cache'

SQL local cache sugerido
CREATE TABLE licencia_cache (
  id TINYINT PRIMARY KEY,
  estado VARCHAR(20) NOT NULL,
  fechaVencimiento DATETIME NULL,
  ultimaValidacionAt DATETIME NOT NULL,
  fuente VARCHAR(20) NOT NULL
);

INSERT INTO licencia_cache (id, estado, fechaVencimiento, ultimaValidacionAt, fuente)
VALUES (1, 'vencida', NULL, NOW(), 'cache')
ON DUPLICATE KEY UPDATE id = id;

Seguridad recomendada
- Usuario remoto de solo lectura:
  - Solo SELECT sobre tabla suscripciones.
- Habilitar SSL hacia Railway MySQL.
- Rotar credenciales cada cierto tiempo.
- No exponer credenciales en frontend.
- Toda validacion de licencias debe quedar en backend.

Operacion diaria
- Para bloquear clinica:
  - UPDATE suscripciones SET estado='suspendida' WHERE clinicaId='X';
- Para reactivar:
  - UPDATE suscripciones SET estado='activa', fechaVencimiento='YYYY-MM-DD HH:MM:SS' WHERE clinicaId='X';

ACLARACION IMPORTANTE: NULL vs VACIO vs 'null'
- NULL (sin comillas) = sin valor real en SQL.
- '' (cadena vacia) = texto vacio, NO es NULL.
- 'null' (con comillas) = texto literal "null", NO es NULL.

Recordatorio de ejecucion: SP crear licencia
- SP: sp_licencia_crear
- Ruta SQL: backend/sql/sp_licencia_crear.sql
- Ejemplo recomendado:
  CALL sp_licencia_crear(
    'DENTALSIVAR-SONSONATE-2026-A8KD-X9PL',
    'DENTAL-SIVAR-SONSONATE',
    'activa',
    'activa',
    1,
    1,
    NULL,
    '2026-12-31 23:59:59'
  );

Notas de fecha
- p_fecha_inicio:
  - Si envias NULL, el SP usa NOW() internamente.
  - Si quieres fecha fija, envia 'YYYY-MM-DD HH:MM:SS'.
- p_fecha_vencimiento:
  - Es la fecha/hora limite de suscripcion.
  - Si ya paso, la validacion responde suscripcion_vencida.
